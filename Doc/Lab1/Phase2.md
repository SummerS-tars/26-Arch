阶段二的核心目标是 **“赋予CPU理解和计算的能力”** 。在这个阶段，信号不仅要在流水线中传递，还要被正确解析（译码 ID）并进行实质性的数学与逻辑运算（执行 EX）。为了在今晚的截止时间前高效推进，我们需要将逻辑梳理得非常清晰，避免在位拼接和符号扩展上栽跟头。
以下是对阶段二的细致讲解与避坑指导：

### 1. 译码阶段 (ID：Instruction Decode) 核心任务
译码阶段的作用就像是“翻译官”，它需要把32位的机器码拆解，并生成后续所有阶段需要的“控制信号”。

- **指令切分 (Instruction Parsing)**：
RISC-V 的一大优势是寄存器索引的位置非常固定。无论什么指令，`rd` 永远在 `[11:7]`，`funct3` 永远在 `[14:12]`，`rs1` 永远在 `[19:15]`，`rs2` 永远在 `[24:20]`。你可以直接用 `wire` 将它们从中抽离出来。
- **立即数生成 (ImmGen)**：
这是极易出错的地方。Lab1 需要处理基础整数指令，主要涉及 I 型（如 `ADDI`）立即数。
  - **避坑指南**：RV64I 是 64 位架构，但指令中的立即数只有 12 位。你**必须**进行符号扩展（Sign Extension），即把最高位（符号位 `inst[31]`）复制填充到高 52 位。
  - *示例逻辑*：`wire [63:0] imm_I = {{52{inst[31]}}, inst[31:20]};`
- **控制信号生成 (Control Unit)**：
建议在 ID 模块内部写一个大的组合逻辑块（如 `always_comb` 或 `always @(*)`），通过 `case(opcode)` 判断指令大类，再嵌套 `case(funct3)` 判断具体指令。
需要生成的关键信号包括：
  - `alu_op`：告诉 EX 阶段的 ALU 具体做什么（加、减、异或等）。
  - `alu_src`：控制 ALU 的第二个操作数是来自寄存器 `rs2` 还是立即数 `imm`。
  - `reg_write_en`：这根线要一直传到 WB 阶段，决定最终是否写寄存器。

### 2. 执行阶段 (EX：Execute) 核心任务
执行阶段就是 CPU 的“算盘”，包含操作数选择和 ALU（算术逻辑单元）计算。

- **操作数选择 (Operand Mux)**：
ALU 通常有两个输入端。A 端一般接 `rs1_data`。B 端则需要一个数据选择器（Mux）：如果 ID 阶段传来的 `alu_src` 为 1，则 B 端接入 `imm`；如果为 0，则接入 `rs2_data`。
- **ALU 核心逻辑**：
根据 `alu_op` 执行具体的加减与位运算。这里可以直接使用 Verilog 的 `+`、`-`、`^`、`<<` 等运算符（再次强调，绝对不要手滑写 `*` 或 `/`）。
- **⚠️ W 后缀指令的特殊处理 (Lab1 核心考点)**：
对于 `ADDIW`、`ADDW`、`SUBW` 等指令，它们的作用是**只对低 32 位进行运算，并将结果的第 31 位（32位结果的符号位）扩展到高 32 位**，最终写回 64 位寄存器。
  - **错误写法**：直接把 64 位的数据相加，然后截断。
  - **正确写法**：先截取源操作数的低 32 位进行运算，得到一个 32 位的临时结果 `res_32`，然后再对其进行符号扩展。
  - *参考逻辑*：`wire [31:0] res_32 = opA[31:0] + opB[31:0];` 然后 `assign res_64 = {{32{res_32[31]}}, res_32};`
