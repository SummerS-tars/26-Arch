# 26-Arch Project Specification

本文档用于定义一个面向当前仓库的通用项目辅助规范，并作为配套 project skill 的人类可读规格说明。

该 Specification 的目标不是讲授泛化的 CPU / FPGA 理论，而是为本仓库中的开发、分析、验证、研究与维护工作提供一个统一、简洁、可更新的项目认知框架。

---

## 1. 目标

该 Specification 与配套 skill 的目标是：

- 为当前仓库建立统一的项目地图
- 为后续 agent / 人类协作提供一致的术语、入口和流程
- 提供标准验证命令与输出判读规则
- 提供面向当前 project 的推荐实践，而不是泛泛而谈
- 提供可低成本维护的结构，使其能随着项目进展持续更新

---

## 2. 适用范围

本 Specification 仅覆盖：

- 当前 `26-Arch` 仓库
- 当前课程 project 的通用知识与规约
- 与本仓库直接相关的命令、验证、文件角色与实现路径

不覆盖：

- 脱离仓库上下文的通用 CPU 教材式知识
- 与本项目无关的泛化 FPGA 工程实践
- 详细到每个阶段所有实现细节的长期设计文档

---

## 3. 设计原则

### 3.1 项目事实优先

所有分析优先依赖仓库中的当前事实，而不是抽象经验：

- `Makefile`
- `README.md`
- `Doc/`
- `ready-to-run/`
- `vsrc/`
- 仿真顶层与接口定义文件

### 3.2 稳定层与易变层分离

为了保证内容可维护，本方案将信息分为两类：

- 稳定层：不频繁变化的项目结构、入口、工作流
- 易变层：当前实现阶段、当前测试结果、当前主要缺口

### 3.3 尽量简洁，但必须准确

该规范强调：

- 不重复低价值背景
- 不堆砌全量终端日志
- 不把“可能如此”写成“当前事实”

### 3.4 服务真实任务

该规范应优先支持以下任务：

- 研究仓库结构和实验主线
- 分析当前实现能做什么、不能做什么
- 运行标准命令并判读输出
- 提炼下一步实现方向
- 低成本更新 skill 与规范本身

---

## 4. 配套 skill 的结构

项目 skill 的建议结构如下：

```text
.agents/skills/26-arch-project-assistant/
├── SKILL.md
├── reference.md
├── verification.md
├── maintenance.md
└── status.md
```

同时提供一份给人读的规格文档：

```text
Doc/Specification/26-Arch-Project-Specification.md
```

各文件职责如下。

### 4.1 `SKILL.md`

职责：

- 定义 skill 何时触发
- 定义默认工作流
- 指导先读取哪些参考文件
- 规定如何控制上下文噪声

要求：

- 尽量稳定
- 不放当前实现状态
- 不直接承载大量项目细节

### 4.2 `reference.md`

职责：

- 保存相对稳定的项目地图
- 记录目录角色、关键入口文件、常用阅读顺序
- 固化项目通用规约

要求：

- 只放“项目事实”与“通用认知”
- 不记录当前某次实现是否通过某测试

### 4.3 `verification.md`

职责：

- 固化标准命令
- 固化成功/失败信号的判读方式
- 提供输出压缩与关键信息提取规则

要求：

- 明确什么是高信号输出
- 明确什么是应折叠的构建噪声

### 4.4 `maintenance.md`

职责：

- 指导如何高效更新整个 skill
- 区分何时改 `SKILL.md`，何时只改参考文件
- 提供最小回归检查方式

要求：

- 强调最小改动原则
- 避免每次项目推进都重写 skill 主文件

### 4.5 `status.md`

职责：

- 提供当前项目阶段和实现效果的高层快照
- 便于人和 agent 快速跟进当前进展

要求：

- 简约
- 准确
- 清晰
- 只保留当前阶段、当前验证边界、主要缺口、下一方向

## 5. 本 project 应纳入的知识域

本 project skill / Specification 应纳入以下高价值知识域。

### 5.1 项目定位与工程主线

包括：

- 该仓库不是空白 RTL 仓库，而是课程提供了仿真、Difftest、测试程序、Verilator 和 Vivado 路径的完整工程框架
- 学生主要在既有框架下实现并扩展 CPU

### 5.2 仓库结构与关键入口

包括：

- `vsrc/`
- `ready-to-run/`
- `difftest/`
- `verilate/`
- `vivado/`
- `Doc/`
- `docs/`
- `Makefile`

以及：

- `core.sv`
- `common.sv`
- `SimTop.sv`
- `VTop.sv`
- `mycpu_top.sv`

### 5.3 Lab 能力演进的高层边界

不要求写成严格课程手册，但至少应支持判断：

- 基础 `Lab1` 主要覆盖什么
- `lab1-extra` 往往意味着什么能力增强
- `Lab2` 开始强调什么能力
- 不能因为仓库里存在测试就默认当前实现已支持该阶段

### 5.4 标准验证流程

包括：

- 应优先运行哪些标准命令
- 每条命令通常用于什么目的
- 如何识别 `GOOD TRAP`
- 如何识别 Difftest 失配
- 如何区分构建失败与运行失败

### 5.5 输出提炼与噪声控制

该部分非常重要，因为它直接影响 agent 使用上下文的效率。

必须支持：

- 保留真正关键的日志行
- 折叠重复 Verilator 构建信息
- 提取：
  - pass/fail
  - failing pc
  - mismatch register / value
  - commit trace 关键片段
  - 下一步建议

### 5.6 推荐实践方向

以本 project 为中心，推荐：

- 解耦模块
- 顶层控制时序、子模块负责组合逻辑
- 先建立骨架，再补功能，再补验证
- 用测试边界反推当前支持范围
- 把项目事实与当前实现状态分开陈述

---

## 6. 标准工作流

### 6.1 仓库研究任务

当用户要求研究仓库、理解 Lab、找入口文件时，应：

1. 先读取项目地图和状态快照
2. 确认关键入口文件
3. 说明项目结构、主线与阅读顺序

### 6.2 当前实现分析任务

当用户要求分析当前 RTL / Lab 实现时，应：

1. 先读取 `status.md`
2. 再看对应实现文件
3. 明确区分：
   - 仓库具备的测试能力
   - 当前实现实际支持的能力

### 6.3 验证与调试任务

当用户要求运行测试、判读输出时，应：

1. 使用标准命令
2. 只保留高信号输出
3. 把失败归因到最可能的子系统
4. 给出最小下一步

### 6.4 规范/skill 更新任务

当项目推进后，应优先：

1. 更新 `status.md`
2. 再判断是否需要更新 `reference.md` / `verification.md`
3. 仅在 skill 行为本身变化时修改 `SKILL.md`

---

## 7. 输出规范

当该 skill 用于实际项目辅助时，输出应满足：

- 简洁优先
- 结论明确
- 区分事实、推断和建议
- 不转述整段无用构建日志

针对测试输出，建议统一压缩为：

```text
Command: <command>
Result: PASS | FAIL
Key line: <success or first failure line>
Relevant detail: <pc / register / instruction / trap>
Next step: <most likely action>
```

---

## 8. 维护原则

### 8.1 小步更新

优先更新最小必要文件，不要因为当前实现进展改变就重写整个 skill。

### 8.2 易变信息下沉

当前通过哪些测试、当前主要缺口、当前下一方向，应放在 `status.md`，不应散落在多个文件中。

### 8.3 事实改动优先修正来源层

例如：

- 目录或角色变化 -> 更新 `reference.md`
- 命令或日志模式变化 -> 更新 `verification.md`
- 当前阶段变化 -> 更新 `status.md`
- 触发逻辑变化 -> 更新 `SKILL.md`

### 8.4 更新后快速回归

更新后至少要确认：

- 路径还存在
- 命令还匹配 `Makefile`
- 当前状态文件不过时
- skill 仍然把稳定层与易变层分开

---

## 9. 当前快照的角色

本 Specification 不应不断记录当前实现细节。

当前项目的高层进展应由：

```text
.agents/skills/26-arch-project-assistant/status.md
```

负责维护。

这样可以保证：

- 规范本身保持稳定
- skill 可持续使用
- 人和 agent 都能快速知道“当前做到哪一步”

---

## 10. 总结

本 Specification 的核心不是“把所有知识都写进去”，而是建立一个对本仓库足够准确、足够简洁、且能够持续更新的项目辅助结构。

这个结构需要同时满足：

- 对当前 project 有足够强的针对性
- 对后续开发与研究足够实用
- 对验证和日志判读有明确规则
- 对后续维护具有低成本更新能力

因此，本 project skill 的正确形态应是：

- `SKILL.md` 负责触发与工作流
- `reference.md` 负责稳定项目知识
- `verification.md` 负责标准验证与输出提炼
- `maintenance.md` 负责高效更新
- `status.md` 负责当前阶段快照
- 本文档负责提供统一的人类可读规格
