### 2. 常见坑点与基础 Debug 思路

#### 坑点 1：Diff-Test 报 PC 错位或寄存器 Mismatch

- **现象**：仿真刚开始跑几条指令，立刻报错 `PC mismatch! Core PC: 0x80000010, NEMU PC: 0x8000000c`。
- **根因**：
  1. **提交逻辑错误**：你把冲刷产生的气泡（NOP）当成了真实的指令提交给了 Diff-Test 接口。NEMU 看到你提交了一个 `addi x0, x0, 0`，于是它的 PC 前进了一步；但紧接着你真正的那条指令来到 WB 级时，NEMU 已经跑飞了。
  2. **没有杀干净**：跳转后，应该被杀掉的两条指令（IF 和 ID 级）只有一条被改成了 NOP，另一条“漏网之鱼”被执行并提交了。
- **Debug 思路**：
检查 `testbench.cpp` 中触发比对的条件（通常是通过传入的 `wb_valid` 信号）。在你的 RTL 中，确保 `wb_valid` 的生成逻辑是：`wb_valid = (当前不是气泡) && (wb_pc != 0) && (!reset)`。如果冲刷了某阶段寄存器，必须将其内部记录的 `valid` 位也置为 0，并随着流水线传到 WB 级。

#### 坑点 2：仿真完美通过，FPGA 上板直接跑飞（死机/亮红灯）
这是架构实验中最令人绝望但也最锻炼人的环节。

- **现象**：`make test-lab3` 提示成功，但烧到板子上后毫无反应，或 LED 乱闪。
- **可能原因与排查思路**：
  1. **时序违例（Timing Violation）**：这是头号嫌疑人。Lab 3 加入了分支判定逻辑。如果你的比较器逻辑写得过于复杂，WNS 变成了负数（比如 -1.5ns），在真实物理电路中，信号还没稳定就被时钟沿打入寄存器了，导致不可预知的随机乱跳。
    - **解决**：查看 Vivado Timing Report。如果是负数，尝试降低 Vivado 中的主频（修改 `clk_wiz` IP，比如从 100MHz 降到 50MHz）；或者优化 RTL 逻辑，将分支计算提前到 ID 阶段。
  2. **未初始化变量（Uninitialized Variables）**：Verilator 仿真器在复位时，通常会自动把所有的 `logic` 信号初始化为 0。但真实的 FPGA 晶体管在上电时状态是**随机的（X）**。
    - **排查**：全局搜索你的代码，确保所有时序逻辑（`always_ff`）在 `if (reset)` 分支下，对**所有的**内部寄存器（特别是流水线段寄存器）都进行了严格的赋 0 操作。绝对不能有遗漏。
  3. **BRAM 延迟不匹配**：在仿真环境中，我们经常假设内存是 0 周期延迟取出的（异步读）。但在真实 FPGA 中，BRAM 通常具有 1 到 2 个周期的同步读取延迟。
    - **排查**：仔细检查 `vivado/src/with_delay` 下的包装器文件，确认板级系统的取指和访存时序是否需要 CPU 插入额外的 Stall 等待周期。

#### 坑点 3：跳转目标的计算精度溢出

- **现象**：在包含极其庞大的代码段的测试用例中（如复杂的排序算法），执行远距离后向跳转（`jal` 或 `bge`）时跳到了错误的地方。
- **根因**：跳转立即数的符号扩展（Sign-Extension）做得不对。比如向后跳时，立即数是一个负数，如果你在解码阶段只取了位数，没有把最高位的 `1` 铺满到 64 位，负偏移量就会变成正向极大的地址。
- **Debug 思路**：在波形图中，找到 `bge` 对应的 `imm` 信号。如果汇编是向后跳，`imm` 必须是诸如 `0xFFFF_FFFF_FFFF_FFF0` 的样子，绝不能是 `0x0000_0000_0000_FFF0`。
